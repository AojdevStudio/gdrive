<objective>
Add read-through caching to Forms `readForm` and cache invalidation to `addQuestion`, matching the established caching pattern used across Gmail, Calendar, Drive, and Sheets modules.

Purpose: Forms is the only module without caching on its read operation. This creates inconsistent performance behavior and means form metadata is fetched from the API on every call, even when unchanged.

Output: `readForm` uses cache check/set/track/log pattern; `addQuestion` invalidates form cache after mutation; unit tests verify both behaviors.
</objective>

   ```typescript
   const cacheKey = `forms:readForm:${formId}`;
   const cached = await context.cacheManager.get(cacheKey);

     context.performanceMonitor.track('forms:read', Date.now() - context.startTime);
     return cached as ReadFormResult;
   ```

2. Replace the existing `context.performanceMonitor.track(...)` call (currently on line 73) and the bare `return` block (lines 75-82) with a block that builds a `result` variable, caches it, tracks performance, logs, and returns:
   ```typescript
   const result: ReadFormResult = {
     formId: response.data.formId || undefined,
     title: response.data.info?.title || undefined,
     description: response.data.info?.description || undefined,
     publishedUrl: response.data.responderUri || undefined,
     editUrl: `https://docs.google.com/forms/d/${formId}/edit`,
     questions,
   };

   await context.cacheManager.set(cacheKey, result);
   context.performanceMonitor.track('forms:read', Date.now() - context.startTime);
   context.logger.info('Retrieved form', { formId, title: result.title });

   return result;
   ```
